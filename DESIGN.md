# 早晨日程 Agent — 设计思路与难点解析

---

## 一、为什么要做这个东西

每天早上起来第一件事，是想清楚今天要干什么。这件事看起来简单，但有两个问题：

**问题一：固定日程和临时任务混在一起很难安排**

比如今天 14:00 有个会，还有三件待办事项要塞进去。人工安排需要不断估算"这个任务多久、那个间隙够不够"，耗费不少脑力，而且容易漏或者排得过满。

**问题二：要有人帮你主动推**

如果只是写个脚本生成排期文件，你还得记得打开看。推到微信才是真正"不需要主动找"的提醒方式，手机弹出来就看到了。

所以目标很清晰：**每天早上自动分析我的日程，把排期结果推到微信，不需要我做任何操作。**

---

## 二、为什么这么设计

### 2.1 为什么用 Agent 而不是普通脚本

最直接的方案是：读文件 → 拼字符串发给 LLM → 拿结果推送。写死流程，简单粗暴。

但这个方案有个问题：LLM 只能被动接收文本，无法主动做事。比如"顺手创建明天的模板文件"这个动作，用普通脚本就得在主流程里硬编码，耦合在一起。

用 Agent 的好处是：**LLM 自己决定什么时候调用哪个工具**。你只需要告诉它"你有这两个能力：读今天日程、建明天模板"，它会在分析完排期后自己去调用建模板的工具，不需要你在代码里写死顺序。

未来如果要加新能力（比如查天气、读 Google Calendar），只需要加一个工具函数，Agent 会自己知道什么时候用，主流程代码完全不用改。

### 2.2 为什么用 Markdown 文件而不是数据库或日历 API

三个原因：

1. **零门槛编辑**：手机备忘录、电脑任何编辑器都能改，不依赖特定 App
2. **LLM 天然理解 Markdown**：`## 固定日程` 这种结构，LLM 解析准确率极高，不需要写解析代码
3. **简单可靠**：文件系统不会挂，不需要网络，不需要授权

代价是需要手动填写，但每晚花 2 分钟填明天的模板是可接受的。

### 2.3 为什么用 FastAPI + APScheduler 而不是系统 cron

系统 cron 直接调 Python 脚本也能实现定时，但有两个缺点：
- 没有手动触发接口，调试必须等到 7:00
- 日志分散，不好追踪

FastAPI + APScheduler 的好处：
- `POST /trigger-review` 随时手动触发，开发调试极方便
- 进程常驻，日志集中在一个终端窗口里
- 以后想加其他接口（比如查历史记录、临时修改触发时间）直接加路由就行

### 2.4 为什么选 GLM-4-Flash

只有一个理由：**永久免费**。

每天触发一次，调用量极低，GLM-4-Flash 的免费额度完全覆盖。对于这种低频、轻量的个人工具，付费模型完全没必要。

## 三、设计流程是怎么走的

```
明确目标
  ↓
拆分职责（每个模块只做一件事）
  ↓
定义接口（模块之间怎么通信）
  ↓
从内到外实现（先工具层，再 Agent 层，再调度层）
  ↓
测试验证（每层独立可测）
```

**具体顺序：**

1. **tools.py**：最底层，纯文件操作，不依赖任何外部服务，最容易写和测试
2. **notify.py**：独立的推送模块，只管发消息，不关心消息内容从哪来
3. **agent.py**：中间层，把工具和 LLM 组装在一起，定义 AI 的行为规则
4. **main.py**：最外层，把所有模块串起来，加上定时触发和 HTTP 接口

这个顺序的好处是：**每一层写完都可以独立验证，不用等整体搭完才能测**。tools.py 写完就能跑测试，不用等 Agent 和 FastAPI 都好了才发现文件读取有 bug。

## 四、中间遇到的难点

### 难点一：langchain-zhipuai 是个空包

按照文档，智谱 AI 的 LangChain 集成应该用 `langchain-zhipuai` 包，里面有 `ChatZhipuAI` 类。但实际安装后发现这个包的 0.0.1 版本是个**空包**，`__init__.py` 里什么都没有，`ChatZhipuAI` 根本不存在。

**解决方案**：换用 `langchain-community` 包，它内置了智谱 AI 的集成（`langchain_community.chat_models.ChatZhipuAI`），这才是真正可用的路径。同时把 `requirements.txt` 里的 `langchain-zhipuai` 替换掉，避免后续误导。

### 难点二：LangGraph API 参数名版本不一致

`create_react_agent` 用于传入 system prompt 的参数，在不同版本里名字不一样：
- 旧版本叫 `messages_modifier`
- 中间版本叫 `state_modifier`
- 新版本（0.2.28+）改叫 `prompt`

直接用 `state_modifier` 在新版 langgraph 上会报 `TypeError`。

**解决方案**：检测实际安装的版本，最终改用 `prompt=SYSTEM_PROMPT`，兼容当前环境。

### 难点三：.env 文件变成了 macOS Alias 文件

用 Finder 的"复制"或某些方式复制 `.env.example` 时，macOS 可能生成 Alias（快捷方式）而不是真正的文件副本。python-dotenv 尝试读取时遇到非 UTF-8 字节，抛出 `UnicodeDecodeError`。

**解决方案**：删掉 Alias 文件，用代码直接写入真正的文本文件。

### 难点四：异步与阻塞的边界

FastAPI 和 APScheduler 都运行在同一个 asyncio 事件循环里。如果任何地方混入了同步阻塞调用（比如用同步版 SMTP 库发邮件），会卡住整个事件循环，导致 FastAPI 无法同时处理其他请求，APScheduler 的其他任务也会被阻塞。

**解决方案**：所有 IO 操作全部使用异步库：
- 邮件：`aiosmtplib`（后来删掉了邮件功能）
- HTTP 请求：`httpx.AsyncClient`
- LLM 调用：`agent.ainvoke`（异步版本）
- 多个推送并发：`asyncio.gather`

### 难点五：测试如何 mock 外部依赖

测试时不能真的发微信消息、不能真的调 LLM API。但 Python 的 mock 有个容易踩的坑：**patch 的路径必须是被测模块的命名空间，而不是原始定义的模块**。

举例：`main.py` 里写的是 `from notify import send_wechat_message`，测试时应该 patch `main.send_wechat_message`，而不是 `notify.send_wechat_message`。如果 patch 错路径，mock 不会生效，测试会真的去发请求。

**解决方案**：统一规则——patch 路径跟着 import 语句走，用 `main.xxx` 而不是 `notify.xxx`。

## 五、设计上的取舍

| 决策 | 选了什么 | 放弃了什么 | 原因 |
|---|---|---|---|
| 日程存储 | Markdown 文件 | 数据库、日历 API | 零依赖，手机可编辑 |
| LLM | GLM-4-Flash | GPT-4、Claude | 免费，够用 |
| 调度方式 | APScheduler 内嵌 | 系统 cron | 有手动触发接口，调试方便 |
| 通知渠道 | 仅 Server酱微信 | 邮件 | 邮件配置复杂，微信更直接 |
| Agent 框架 | LangChain + LangGraph | 自己写 HTTP 调用 | 工具扩展方便，未来加能力不改主流程 |

---

## 六、一句话总结设计哲学

> 用最少的代码、最少的依赖、最少的配置，做到"每天早上不用动脑，打开微信就知道今天怎么过"。
